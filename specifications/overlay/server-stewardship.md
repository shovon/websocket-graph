# Server Stewardship: Discretionary Inspection for Stability

## Status of This Memo

This document is **non-normative**. Like the base architecture (`architecture.md`), the path-emergence memo (`path-emergence.md`), and the hardening memo (`hardening-the-overlay.md`) it sits beside, it specifies nothing, requires nothing, and conforms to nothing. It is a rationale memo. Its subject is a single latitude the architecture leaves the server — the server is under no obligation to look inside the messages it relays, but it is nowhere forbidden from doing so — and its argument is that this unspent freedom is exactly the lever a path-emergence overlay needs to keep itself stable, provided the server spends it on the one thing it is allowed to and not on the one thing it is not.

Read it for the *why*. When a normative companion someday fixes what an inspecting server may meter, refuse, or evict on, that document carries the requirements; this one carries the reasoning behind the shape they will take. Nothing here uses RFC 2119 keywords, and where it appears to recommend, read "this is the judgment a stability-minded deployment should weigh," not "you must." The register is the discursive one this corpus reserves for rationale: it argues, it hedges, it repeats a load-bearing point on purpose.

Section references resolve by name, not by location. Those of the form (Hardening §N) point into `hardening-the-overlay.md` and (Emergence §N) into `path-emergence.md`, and (Completion §N) into `path-completion.md` — companion overlay memos in this same folder. Those of the form (Architecture §N) point into `../architecture.md`, (Relay §N) into `../relay-and-neighborhood-semantics.md`, and (Payload §N) into `../payload-string.md` — substrate documents that sit one level up from this folder.

## Abstract

The relay carries an opaque payload it never interprets (Payload §4), and the architecture is built around a server whose part is deliberately small (Architecture §1). It is tempting to read that smallness as a prohibition — that the server *must not* look at what it carries. It is not. "Need not" is not "must not." The server is the one trusted party in the system (Architecture §8); it terminates every edge, sees every send, and already holds sole authority over who exists in the graph (Architecture §4). The freedom to inspect what crosses it is simply unspent, and this memo is about spending it well.

The case for spending it is that a path-emergence overlay (`path-emergence.md`) has a hole no endpoint can close. The overlay's untrusted-carrier stance buys its endpoints *non-deception* but explicitly not *availability* (Hardening §4): a node that floods abusively or quietly swallows everything it should forward degrades every trail that crosses it, and no endpoint can either localize it — silence is ambiguous (Completion §3, §5) — or remove it. The server can do both. It sits where the behavior is visible and holds the one lever, eviction, that no participant has.

The case for spending it *carefully* is that the architecture draws a bright line (Architecture §7.1): a server that makes **relay decisions** from a payload's contents is not a GRS derivative but a different protocol. So the whole discipline of this memo is one distinction. Inspection may inform **membership** — whether a node stays in the graph — and must never touch **routing** — where an accepted message goes. The payload's content, or a node's conduct, may change whether a node continues to exist; it must never change where a message is relayed. Held to that line, server stewardship adds nothing a client can even observe except the disappearance of abusers, which the base already permits at any moment — and it turns out the most valuable signals are behavioral and need no payload inspection at all.

## 1. The freedom, and the floor under it

Begin with what the base actually says, because the freedom is easy to mistake for its opposite.

The server relays an opaque payload. The string a node sends is carried verbatim and never interpreted (Payload §4); the routing that matters to the server is done entirely on the *envelope the server owns* — the designator, resolved against the sender's current neighborhood (Relay §3, §4) — and not on the bytes inside. The architecture leans on this everywhere: the server's part is "deliberately small" (Architecture §1), it relays "just enough" and leaves the substance to clients (Architecture §3.2), and it can stay blind to content with no loss to any guarantee it owes. A server that never once looks inside a payload is a perfectly good GRS server. For privacy it is arguably the *best* GRS server (Section 6). Blindness is the comfortable default, and nothing here disturbs it as a default.

But blindness is a default, not a duty. Nowhere does the base require the server to remain ignorant of what it carries. And three facts about the server make its freedom to look more than a technicality:

- **It is trusted by assumption.** GRS already grants the server authority over the entire graph and already concedes that it can read all client-to-client traffic (Architecture §8). There is no confidentiality boundary at the substrate for inspection to *breach*; the overlay erects its own, with encryption (Hardening §5), and that is the boundary that matters — more on the interplay in Section 6. The point for now is narrow: inspection crosses no line the base drew.
- **It is the one chokepoint.** Every send, on every edge, in every flood, passes through the server — flooding *is* individual sends to each out-neighbor (Emergence §4), so even a network-wide flood is, from the server's seat, a burst of ordinary relays it personally performs. Whatever behavioral truth there is to be had about a node, the server is the one party positioned to have it.
- **It already governs membership.** The server's stated responsibilities already include detecting departed or unreachable nodes and repairing the graph around them (Architecture §4), and the security considerations already name connection-churn DoS as something implementations should resist (Architecture §8). Deciding who remains in the graph is not a new power this memo invents; it is a power the server has from the first page. Inspection only gives that existing power better-informed input.

So the freedom is real and the floor under it is solid: the server *may* look, it is uniquely placed to, and acting on what it sees is continuous with a job it already has. The only question is what it is allowed to *do* with the result — and that is where the discipline lives.

## 2. The line that must not be crossed

The architecture forbids exactly one thing here, and it forbids it sharply: a derivative that "has the server make relay decisions from a payload's contents is not a GRS derivative but a different protocol, and MUST NOT claim conformance" (Architecture §7.1). Everything in this memo lives or dies by respecting that sentence, so it is worth being precise about what it does and does not rule out.

The server has two quite different verbs, and the prohibition lands on only one of them.

**Relay** is the per-message verb: given a send the server has accepted, *which* out-neighbor receives it, and *whether* it is delivered. The base fixes this completely. The target is the designator resolved against the sender's *current* neighborhood; the message goes to the denoted out-neighbor or is discarded (Relay §3, §4); No-Misdelivery holds. This decision is a function of the designator and the live graph, and of nothing else. The moment it becomes a function of payload content — "this one looks like spam, send it elsewhere"; "this nonce I dislike, divert it" — the server is routing on content, and that is the forbidden protocol. Inspection must never feed this verb. Not to redirect, not to selectively misdeliver. The relay stays content-blind even on a server that reads every byte for other purposes.

**Membership** is the structural verb: whether a node exists in the graph at all, where it is placed, and when it is removed and the graph repaired around it. This verb is *already* the server's, already content-agnostic in the base, and already exercised whenever a client disconnects or is judged gone (Architecture §3, §4). Nothing in the base says the server may decide membership only from connection liveness; it says the server is responsible for keeping the graph healthy and repairing it. Letting observed conduct — *including* conduct read from a payload — inform whether a node keeps its place is a membership decision, not a relay decision. The payload changed *who is in the graph*, never *where a message went*.

The whole reconciliation is that one assignment: **content may move a node out of the graph; content may never move a message to a different edge.** Inspection is allowed to feed the structural verb and forbidden to feed the per-message one.

And the reason this is not a lawyer's loophole but a genuinely conformant position is **substitutability** (Architecture §7.1). The test of a derivative is that code written to the base keeps working unmodified against it. Ask what any *remaining* node observes when the server evicts an abuser: a node that was its out-neighbor is gone, and its neighborhood has been repaired. That is indistinguishable from the node having simply disconnected — which the base already says may happen at any instant, after which the server may even relocate the survivors anywhere in the graph (Architecture §3). Eviction-for-misbehavior produces *only graph states the base already permits at any time for any reason or none*. A client cannot tell a policing server from a blind one, because policing manufactures no state that ordinary churn could not. The relay contract every client relies on — best-effort delivery along current edges, No-Misdelivery, resolution against the live neighborhood — is untouched. That is why this is stewardship within GRS and not a fork of it.

## 3. The hole in the endpoint stance that only the server can fill

Why reach for this freedom at all? Because path-emergence has a specific, structural gap, and it is the server-shaped one.

The overlay's underlying posture is the untrusted-carrier stance (Hardening §4): every relaying node is a conduit whose honesty you assume nothing about, and trust lives only end to end. That stance is magnificent at what it does — it collapses every relay failure mode, "crashed, lazy, buggy, curious, or actively malicious," into one indistinguishable event: *my authenticated answer did not come back.* But read the fine print the hardening memo itself insists on: the stance delivers **non-deception, not delivery**. It guarantees you will never be *fooled* into thinking a dropped message arrived. It does exactly nothing to make the message actually arrive past a carrier that declines to carry it.

Now layer path-emergence on top and watch where that bites. A trail threads specific nodes (Completion §4). Suppose one of them is a black hole — it accepts what it is handed and forwards nothing. The endpoints discover this in the only way the stance allows: an acknowledgement that never comes (Completion §3, §5). And then they are nearly helpless about it. Silence is ambiguous — a dead trail, a dropped ack, a departed peer, or a malicious swallow all look identical (Completion §3) — so the endpoints cannot even *name* the culprit, let alone evict it. Their remedies are the honest, limited ones the memo lists: back off, and fail over to a spare trail (Completion §5). Those route *around* the bad node. They never *remove* it. It stays in the graph, swallowing every flood and every trail that happens to cross it, taxing the whole network indefinitely, and each pair of endpoints rediscovers its badness the hard way and pays again to route around it.

The same asymmetry haunts the other direction. Path-emergence opens by naming the broadcast-storm failure mode — many nodes flooding many destinations until the network carries only discovery and no content (Emergence §1) — and its defense is a *voluntary* discipline: back off before re-flooding, longer each time (Completion §5). Voluntary is the operative word. A misbehaving node need not obey its own back-off. A flood-happy or actively hostile node can re-search in a tight loop, and no endpoint has any lever to make it stop; the discipline binds only the well-behaved. The hardening memo is candid that this whole class — "flood-amplification rate-limiting and proof-of-work," "selective-drop and black-hole detection," Sybil resistance — is *deferred* work, parked because the end-to-end signature made it safely additive (Hardening §6). Deferred is not free; it is a hole, held open on the promise that something can fill it later without redesign.

The server is the something. It is the one party that can do *both* halves of what the endpoint stance cannot: it can **see** cross-node behavior the endpoints can only infer (it watches a node receive a steady stream and forward nothing; it watches a node flood ten times a second), and it can **act** by the one mechanism no participant possesses — taking the node out of the graph. So server stewardship is not a new security model bolted beside the overlay's. It is a *realization, at the center, of items the hardening memo already filed under "defer freely"* (Hardening §6) — the cooperation-and-availability defenses — placed at the one vantage that can enforce them. It sits squarely in the deferred bin. It fights only denial, never the integrity the end-to-end signature already nailed down. It is exactly the kind of additive that the pivot in Hardening §3 was designed to make possible.

## 4. What the server can see — and what encryption hides

Here is the pleasant surprise, and it sharpens the user-facing slogan "the server may inspect a relayed message." The server's most valuable stability signals are not in the message at all. They are in the *conduct*, and the conduct is visible without opening the envelope.

Sort what the server can know into two bins.

**Behavior — visible from the relay's own bookkeeping, no inspection required.** The server performs every send personally, so without reading a single payload byte it already knows: how often a node sends, and to how many out-neighbors at once (a flood is a fan of individual sends — Emergence §4, so a flood *looks like* a flood from the server's seat); whether a node that receives a steady inflow ever emits a corresponding outflow, or sits as a sink (the black-hole signature); how a node churns its connection (the §8 DoS signal). None of this is "the contents of a payload" in the sense Architecture §7.1 guards. It is metadata the server generates by doing its job. This is the richest and safest vein, and it is worth saying plainly: **the cooperation signals that matter most are behavioral, and reading messages is not how you get them.** A node's cooperation is a question about whether it forwards, not about what it forwards.

**Content — visible only to the extent the overlay leaves it visible.** This is inspection proper: opening the payload to read the envelope. And here the overlay deliberately works against the server, which is the whole point of Hardening §5. The sealed core — origin, target, nonce, budget, type, content — is end-to-end and, where confidentiality is in scope, encrypted to the recipient; path-emergence goes further and has `B` seal the ferried forward trail to `A` specifically, so even the route travels in the dark (Completion §6). So the semantic signals one might *wish* to police on — a node re-injecting a duplicate `(origin, nonce)` to amplify a flood, a node tampering with an accumulated trail — are frequently behind encryption the server cannot pierce. Content inspection yields whatever the overlay left in the clear (perhaps a layer's type marker, perhaps an unencrypted transit annotation) and no more.

Two things follow, and they point the same way.

First, **lean on behavior, not content.** It is conformance-comfortable — behavior-based metering and eviction never read a payload's contents, so they cannot even be accused of feeding the forbidden relay verb (Section 2); throttling by *volume* is not a content decision at all. And it is robust: a signal the server derives from a node's observable conduct cannot be defeated by the node *lying* in a payload the server was never going to read. A black hole cannot encrypt away the fact that nothing comes out of it.

Second, **be honest that this is policing the obvious, not a guarantee.** A clever adversary forwards *just enough* to look alive while dropping selectively; a Sybil swarm (Hardening §2, §6) spreads its abuse thin across many minted names; encryption hides the semantic tells. The server's inspection catches the loud and the lazy, raises the cost of the subtle, and stops none of the determined cold. That is enough to be worth doing and not enough to lean on — which is precisely the posture Section 7 insists the endpoints keep.

## 5. How it acts: only the levers it already holds

When the server decides a node is a problem, it reaches for nothing new. Both available actions are powers it had before this memo existed.

**Throttle, by declining to relay.** Relay is best-effort; the server "attempts to deliver but does not guarantee" (Architecture §6, Relay §6). Choosing not to relay a node's excess sends is not a violation of the contract — it is the no-guarantee latitude the contract already grants, exercised by *volume* rather than by content. A node flooding past any plausible back-off (Completion §5) can have its surplus sends simply dropped. This is the gentle lever: it caps a storm without ending a node, and because it keys on send-rate and fan-out it never touches a payload's contents.

**Evict, and repair.** The structural lever (Section 2): remove the node, repair the graph around its absence — the exact machinery the base already runs when any node departs (Architecture §3, §4). To every surviving node this is a departure like any other. This is the lever for the unrepentant: the persistent black hole, the flood engine that ignores throttling, the churn attacker.

A sober deployment escalates between them rather than jumping to eviction — throttle first, evict on persistence — and treats its own judgments as fallible, since Section 4 just admitted the signals are partial and evadable. The cost of a wrong eviction is borne by an honest node wrongly cut from the graph; the base lets the server relocate and repair freely, so the *graph* recovers instantly, but the *node* is gone, and a deployment should weigh false positives accordingly. None of this needs new protocol surface, which is the tell that it stays inside GRS: the server is using its old verbs with better-informed inputs.

## 6. The cost: observation concentrated in the trusted party

Stewardship is not free, and the price is paid in exactly the currency this corpus is most careful about. So state it squarely rather than bury it.

The architecture already concedes the server can read all traffic and names server compromise as a top risk (Architecture §8). Inspection does not *create* that exposure — it is latent the moment you have a relaying server — but it *institutionalizes* it. A deployment that inspects has built the observation apparatus, normalized its use, and made profiling of node behavior a running function rather than a dormant capability. That is more traffic-analysis power, made routine, concentrated in the single party an attacker most wants to subvert. It is the same hazard the hardening memo flagged as the one deferral that *leaks while you wait*: relationship-hiding, where a cleartext "`a` wants `b`" hands every observer your social graph (Hardening §6). An inspecting server is precisely such an observer, sitting at the best seat in the house.

The thing that *bounds* the cost is the overlay's own encryption, and this is where the two designs meet in a deliberate, almost adversarial, equilibrium. The overlay encrypts to limit what any carrier — the server included — can learn (Hardening §5, Completion §6). The server inspects to police the overlay's carriers. They settle at a stable line: **behavior is visible, content is sealed.** The server can tell that a node floods and that a node swallows; it generally cannot tell *what* the flood seeks or *who* the swallowed message was for, if the overlay did its sealing. That equilibrium is healthy. It lets the server do the cooperation policing only it can do, while denying it — and a future compromise of it — the semantic surveillance the overlay exists to prevent. A deployment that wants the policing should *want* the encryption too; they are not in tension so much as in balance, each defining the other's reach.

The conclusion is a posture, not a mandate. Stewardship is a discretion a deployment *chooses*, with a real privacy-and-centralization price, and different deployments will rightly sit at different points on the line from fully blind to actively policing. It is a *may*, weighed, never a *should*, assumed. A blindness-preferring deployment that leans entirely on the endpoints' own back-off and multipath is making a coherent, conformant, privacy-favoring choice; a stability-pressed deployment that polls behavior and evicts abusers is making a coherent, conformant, availability-favoring one. The memo's job is to make sure that whichever a deployment picks, it picks it on purpose and within the line of Section 2.

## 7. What this is, and what it is not

Two sentences carry the whole memo, and they are worth ending on because mistaking either one undoes it.

**What this is:** an additive, optional realization — at the one vantage that can enforce it — of the cooperation-and-availability defenses the hardening memo always meant to defer (Hardening §6). It fills the hole the untrusted-carrier stance leaves open by structural necessity (Section 3), using only the server's existing authority over membership (Section 2) and only the signals its existing role already exposes (Section 4). It manufactures no graph state ordinary churn could not, so it costs clients no compatibility (Section 2). It is the server, at its own discretion, thinning the herd of obvious abusers so that the honest majority's trails stay cheap.

**What this is not:** a reason to weaken anything at the endpoints. The overlay must keep treating *every* carrier — the policed network included — as untrusted (Hardening §4). Path-emergence's machinery stays exactly as written: the end-to-end signatures, the candidate-until-proven confirmation folded into traffic, the acknowledgement-or-silence liveness test, the back-off, the multipath spares (Completion §3, §5). Server stewardship catches the loud and the lazy and raises the price of the subtle; it stops the determined adversary not at all, and it goes blind wherever the overlay encrypts (Section 4). The server does not have your back. It thins the obvious, and the obvious is worth thinning — but an overlay that relaxes its own defenses because "the server is watching" has traded a guarantee it held for a favor it was never promised. Keep the endpoint stance whole, let the server steward what only it can, and the two together carry path-emergence further than either alone — which was the point of spending the freedom at all.
